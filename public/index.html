Features

Built for practical gateway maintenance

Keep the upstream secret on the server while giving users controlled, OpenAI-compatible access.

API

OpenAI-style API

Supports `/v1/models`, `/v1/responses`, and `/v1/chat/completions`.

KEY

Downstream keys

Users receive gateway keys, while your upstream API key remains server-side.

CAP

Quotas and limits

Control models, endpoints, streaming, RPM, token caps, and budget by group.

LOG

Admin visibility

Inspect masked call logs and block problematic keys without exposing raw secrets.

Quick start

Run locally with Docker

PowerShell 
git clone https://github.com/zhaozehan0424-design/responses-api-gateway.git
cd responses-api-gateway
Copy-Item .env.example .env
# edit .env with your upstream base URL, upstream key, and downstream keys
docker compose up -d

Configure upstream

Set `UPSTREAM_API_BASE` and `UPSTREAM_API_KEY` in server-side env vars.

Create user keys

Add simple `DOWNSTREAM_API_KEYS` or configure group-based key policies.

Point clients to `/v1`

Use `http://localhost:4000/v1` or your Vercel domain as the base URL.

Example

Call the Responses API

cURL 
curl https://your-domain.example/v1/responses \
  -H "Authorization: Bearer YOUR_DOWNSTREAM_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "claude-haiku-4-5-20251001",
    "input": "Reply with OK only.",
    "max_output_tokens": 16
  }'
